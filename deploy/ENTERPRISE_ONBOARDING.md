# AI Team OS 企业开通流程

本文件定义企业正式开通的目标流程和当前限制。当前代码没有一个经过平台管理员鉴权、审计、幂等和事务保护的商业 provisioning API，可以安全地创建并绑定 `TenantCompany`、套餐订阅、企业负责人和现有 `Tenant`。因此在该能力补齐前，不得通过手工 SQL 或 Prisma Studio 为付费企业开通；Phase 13 tag 不是商业开通批准。

## 当前数据一致性约束

选择一个不可变的企业标识作为 canonical company key，并保证：

- `TenantCompany.id` 等于 `TeamOrganization.companyId`；
- 所有 Team OS 业务记录的 `companyId` 都来自服务端已验证成员关系，不能信任客户端传值；
- 共享账号/知识链路使用 `Tenant.id` 和 `User.tenantId` 时，必须与 canonical company key 相同，或通过唯一、受审计的映射表转换；
- 一个用户跨企业任职时，企业选择必须由有效 `TeamMember` 关系解析，不能用请求参数自行切换；
- 迁移前后运行一致性检查，禁止出现孤立 companyId 或跨企业关系。

当前组织仓库在部分路径读取 `Tenant` 名称，并在独立账号路径使用 `team-os:<ownerUserId>` 作为 companyId；商业开通前必须确定统一映射策略并用集成测试覆盖，不能依赖人工记忆。

## 正式目标流程

### 1. 企业资料与负责人

平台管理员通过未来的安全 provisioning API：

1. 创建或核对 canonical `Tenant`。
2. 创建同 ID 的 `TenantCompany`，设置企业名称、行业、状态和 `ownerId`。
3. 绑定一个已完成身份核验、状态为 active 的负责人账号。
4. 在同一事务或可补偿工作流中创建首个 `TeamOrganization` 与 `TEAM_OWNER` 成员关系。
5. 写入操作人、请求 ID、幂等键、前后值和时间戳到审计日志。

若任一步失败，整个开通过程必须可重试且不会生成第二个企业或第二个 Owner。

### 2. 套餐与功能

1. 选择已审核的 `SubscriptionPlan`。
2. 创建 `TenantSubscription`，明确开始/结束时间和状态。
3. 核对 `FeaturePermission`、用户数、存储量和 AI 调用限制。
4. 从服务端业务 API 验证受限功能；不能只看套餐页面或功能检查接口。
5. 设置到期/停用演练，确认 `DISABLED`/`EXPIRED` 企业无法继续调用受保护 API。

当前套餐升级接口只记录/返回升级意图，不代表已完成扣款或订阅变更。真实计费、授权与服务端 feature guard 在业务 API 的强制执行是商业上线前置条件。

### 3. 员工与邀请

当前邀请能力只创建一条含 7 天有效 `inviteCode` 的 `TeamInvitation` 记录：

- 没有公开的 invite-code 接受/注册闭环；
- 没有真实邮件、短信或 APP 推送投递；
- 邀请代码应视作凭据，不得写入普通日志或工单正文；
- 被邀请邮箱对应账号必须先注册且保持 active；
- 最终由 `TEAM_OWNER` 在成员管理中按同一邮箱、同一角色添加用户；有效邀请会在添加成功时转为 `ACCEPTED`；
- 跨企业用户必须被拒绝，角色不能与邀请不一致。

企业开通 SOP 必须把“创建邀请”和“成员实际加入”作为两个独立验收点，不能把邀请码生成当作员工已入职。

### 4. 初始化业务能力

按最小权限依次完成：

1. 创建团队并添加 Owner、Manager、Trainer、Member 测试账号。
2. 配置套餐权限，并用服务端拒绝结果验收限制。
3. 初始化企业知识范围，验证知识查询只使用该企业 tenant/company 映射。
4. 创建一条测试任务，完成员工提交与主管查看。
5. 创建一条虚拟 CRM 客户，验证跨企业不可见。
6. 创建培训课程/分配，验证角色权限。
7. 验证数据看板仅汇总当前企业。
8. 验证站内通知创建、读取和已读状态；外部推送当前未完成，不得宣称已送达。
9. 删除或匿名化验收用客户隐私数据，保留审计证据。

## 开通审批清单

- [ ] provisioning API 已上线并通过鉴权、审计、幂等、事务与回滚测试。
- [ ] canonical company key 与 Tenant 映射方案已冻结并通过一致性扫描。
- [ ] Owner 已实名确认，账号 active，未绑定冲突企业。
- [ ] 套餐状态、到期时间、用户/存储/AI 限额正确。
- [ ] 受限功能在服务端业务 API 被拒绝，而非仅 UI 隐藏。
- [ ] 企业 A/B query、body、path 越权测试全部通过。
- [ ] 企业、团队、成员停用状态测试全部通过。
- [ ] RDS 快照、逻辑备份和恢复演练完成。
- [ ] 日志脱敏、告警、隐私与数据保留政策确认。
- [ ] APP 若对外发布，已通过 `APP_RELEASE_READINESS.md`。
- [ ] 业务、安全、运维三方签字并记录 go/no-go 结论。

任一安全项未通过时只能在隔离的 pilot 环境继续测试，不能以 Phase 13 tag 替代上线审批。
