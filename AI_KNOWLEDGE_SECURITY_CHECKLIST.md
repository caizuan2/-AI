# AI 知识库安全检查清单

本文档用于 AI 知识库升级过程中的安全审查。所有新增用户端、管理员端、投喂、文件解析、向量检索和跨端构建能力都必须逐项检查。

## 1. 项目安全边界

- [ ] 不重新创建项目。
- [ ] 不覆盖现有核心代码。
- [ ] 不删除现有文件。
- [ ] 不清空数据库。
- [ ] 不重置配置文件。
- [ ] 不移除现有认证体系。
- [ ] 新功能通过新模块、新路由、新服务、新页面渐进接入。
- [ ] 每次修改后检查 `git status`、`git diff --stat`、`git diff`。
- [ ] 不自动执行 `git commit`。
- [ ] 不自动执行 `git push`。

## 2. 当前项目已识别安全事实

| 项目 | 当前状态 | 升级要求 |
| --- | --- | --- |
| 登录 | 已有 session cookie 登录 | 必须复用 |
| 卡密 | 已有 license gate | 必须复用 |
| 管理员 | 当前通过环境变量和 bootstrap 手机号判断 | 需要增量补 RBAC |
| 用户隔离 | 多数知识接口按 `userId` 查询 | 新接口继续防 IDOR |
| 知识删除 | `/api/knowledge/[id]` 存在物理删除 | 升级时必须收紧 |
| 文件解析 | 已支持 txt/md/pdf/docx | 继续增强校验和任务化 |
| 向量库 | 已使用 pgvector | 新增抽象层时复用 |
| 限流 | middleware 中已有 API rate limit | 新接口必须加入 |
| 安全响应头 | middleware 中已有基础安全头 | 继续保留 |

## 3. RBAC 权限清单

### 3.1 角色定义

- [ ] `user`：只能提问和查看自己的会话。
- [ ] `kb_admin`：可以投喂、查看任务、重试失败任务、查看审计日志。
- [ ] `kb_admin`：不能删除历史文件。
- [ ] `kb_admin`：不能物理删除知识库数据。
- [ ] `super_admin`：可以管理角色。
- [ ] `super_admin`：如需删除，只允许软删除。
- [ ] `super_admin`：所有危险操作必须写审计日志。

### 3.2 认证复用

- [ ] 继续使用 `requireUser()`。
- [ ] 继续使用 `requireLicensedUser()`。
- [ ] 新增 `requireRole()` 时内部复用现有 session。
- [ ] 不改变登录、注册、卡密激活流程。
- [ ] 环境变量 admin 在 RBAC 初始化前可作为 bootstrap `super_admin`。

### 3.3 权限拒绝

- [ ] 用户访问管理员接口返回 403。
- [ ] 未登录访问用户接口返回 401。
- [ ] 未授权访问知识资源返回 404 或 403，避免泄露资源是否存在。
- [ ] `kb_admin` 调用删除接口返回 403。
- [ ] 权限拒绝事件可记录到审计日志。

## 4. 用户端 API 安全清单

目标接口：

```text
POST /api/ai/chat/ask
GET  /api/ai/chat/history
GET  /api/ai/chat/conversations
```

检查项：

- [ ] 必须登录。
- [ ] 必须通过卡密或产品授权。
- [ ] 只能读取当前用户自己的会话。
- [ ] `conversation_id` 必须校验归属权。
- [ ] 附件 ID 必须校验归属权。
- [ ] 不能接受管理员投喂字段。
- [ ] 不能写入知识库。
- [ ] 不能触发知识删除。
- [ ] 请求体大小有限制。
- [ ] 对问题文本长度做限制。
- [ ] 对附件数量做限制。
- [ ] 对 `mode` 做枚举校验。
- [ ] 对 `enable_deep_thinking`、`enable_web_search` 做布尔校验。
- [ ] 响应不暴露内部 storage path。
- [ ] 响应不暴露 API key、数据库连接、provider secret。
- [ ] 响应不暴露调试字段，如 chunk 原始 embedding、内部 score 明细、fallback 堆栈。
- [ ] 无知识依据时禁止编造。

## 5. 管理员端 API 安全清单

目标接口：

```text
POST /api/admin/kb/ingest/text
POST /api/admin/kb/ingest/chat
POST /api/admin/kb/ingest/file
GET  /api/admin/kb/files
GET  /api/admin/kb/jobs
GET  /api/admin/kb/jobs/:id
POST /api/admin/kb/jobs/:id/retry
GET  /api/admin/kb/audit-logs
```

检查项：

- [ ] 所有接口必须登录。
- [ ] 所有接口必须校验角色。
- [ ] `kb_admin` 只允许投喂和查看。
- [ ] `kb_admin` 不能删除文件。
- [ ] `kb_admin` 不能删除 chunk。
- [ ] `kb_admin` 不能 reset。
- [ ] `kb_admin` 不能 clear。
- [ ] `super_admin` 执行软删除必须写审计。
- [ ] 查询列表必须分页。
- [ ] 查询列表必须限制最大 page size。
- [ ] 任务 retry 必须校验任务状态。
- [ ] 任务 retry 必须防止重复并发执行。
- [ ] 所有写操作记录 `userId`、`role`、`ip`、`userAgent`。
- [ ] 错误响应不暴露内部路径和密钥。

禁止暴露给 `kb_admin`：

```text
DELETE /api/admin/kb/files/:id
DELETE /api/admin/kb/chunks/:id
DELETE /api/admin/kb/reset
DELETE /api/admin/kb/clear
```

如果这些接口为系统维护保留：

- [ ] 仅 `super_admin` 可访问。
- [ ] 默认软删除。
- [ ] 必须写审计日志。
- [ ] 必须有二次确认字段。
- [ ] 禁止前端普通管理页面展示入口。

## 6. 文件上传安全清单

### 6.1 通用校验

- [ ] 校验文件大小。
- [ ] 校验 MIME 类型。
- [ ] 校验扩展名。
- [ ] 尽可能校验文件头 magic bytes。
- [ ] 限制单次上传文件数量。
- [ ] 限制单用户单位时间上传次数。
- [ ] 文件名做规范化，不信任原始文件名。
- [ ] 禁止路径穿越，如 `../`。
- [ ] 不把 `storage_path` 直接返回前端。
- [ ] 私有存储路径和公开访问路径分离。
- [ ] 上传失败时清理临时文件。
- [ ] 上传成功后写 `IngestionJob`。
- [ ] 上传成功后写 `AuditLog`。

### 6.2 文件类型

- [ ] TXT / Markdown：限制编码和大小。
- [ ] PDF：限制页数、大小和解析耗时。
- [ ] Word：限制大小和解析耗时。
- [ ] Excel：限制 sheet 数、单元格数量和大小。
- [ ] PPT：限制页数、媒体数量和大小。
- [ ] 图片：OCR provider 未启用时返回明确状态，不硬编码假结果。
- [ ] 音频：ASR provider 未启用时返回明确状态，不硬编码假结果。
- [ ] 视频：视频 provider 未启用时返回明确状态，不硬编码假结果。

### 6.3 解析任务

- [ ] 解析在 job 中执行，避免同步请求超时。
- [ ] 任务状态包括 `pending`、`processing`、`completed`、`failed`、`retrying`。
- [ ] 失败原因保存为可读信息。
- [ ] 失败原因不包含密钥。
- [ ] retry 有次数上限。
- [ ] retry 写审计日志。

## 7. 数据库安全清单

- [ ] 迁移只新增表、字段或索引。
- [ ] 不删除现有表。
- [ ] 不删除现有字段。
- [ ] 不清空数据。
- [ ] 不重置数据库。
- [ ] 不创建和现有 `knowledge_chunks` 冲突的同名表。
- [ ] 使用事务时只放数据库写入，不放网络请求和 embedding。
- [ ] 大批量任务按 item 或小批次处理。
- [ ] 关键查询字段加索引。
- [ ] 列表接口分页。
- [ ] 软删除字段使用 `deletedAt`、`deletedByUserId`。
- [ ] 物理删除只允许维护脚本，且默认不提供。
- [ ] 审计日志不可由普通管理员删除。

## 8. 知识库删除与保留策略

- [ ] `kb_admin` 永远不能物理删除知识库文件。
- [ ] 用户端没有知识删除入口。
- [ ] 管理员端默认隐藏删除按钮。
- [ ] 后端权限拒绝优先于前端隐藏。
- [ ] `super_admin` 如需删除，必须软删除。
- [ ] 软删除保留 `deletedAt`、`deletedByUserId`、`deleteReason`。
- [ ] 删除操作写 `AuditLog`。
- [ ] 删除操作不能级联清空历史审计日志。
- [ ] 向量清理应异步执行，并记录任务日志。

## 9. RAG 与 AI 安全清单

- [ ] 检索只在当前授权知识范围内执行。
- [ ] 不把其他用户知识作为上下文。
- [ ] 不把私有 storage path 交给模型。
- [ ] Prompt 中要求无依据时说明资料不足。
- [ ] 不把系统 prompt 返回给前端。
- [ ] 不把 provider API key 返回给前端。
- [ ] 不把完整内部检索流程返回给普通用户。
- [ ] 对用户问题做长度限制。
- [ ] 对附件抽取文本做长度限制。
- [ ] 对上下文 chunk 数量做上限。
- [ ] 对模型输出做安全清理。
- [ ] 对 prompt injection 做基础防护，知识内容不得覆盖系统指令。
- [ ] AI 失败时返回可读错误，不返回堆栈到生产前端。

## 17. 当前阶段结论

当前项目适合做增量升级，不适合重写。最优先安全动作是：

1. 建立 RBAC。
2. 建立审计日志。
3. 收紧知识删除能力。
4. 将管理员投喂和用户问答拆成清晰 API 边界。
5. 将文件解析和向量化任务化。
6. 在不破坏现有 `/chat` 和知识库功能的前提下逐步替换入口。
