# AI 知识库 APP 项目说明

## 产品目标
开发一个“对话式投喂型 AI 知识库 APP”。
第一版只做闭环：聊天投喂 + AI 自动整理 + 知识入库 + 知识问答 + 引用来源。

## 技术栈
- Next.js App Router
- TypeScript
- Tailwind CSS
- shadcn/ui
- Prisma
- PostgreSQL
- pgvector
- OpenAI API

## 代码规范
- 所有代码使用 TypeScript
- API 返回统一 JSON
- UI 先做简洁可用，不追求复杂动画
- 所有核心功能都要有 loading、empty、error 状态
- 不要一次性做高级功能，比如团队协作、知识图谱、语音、微信导入

## MVP 功能
1. 用户可以在聊天框中投喂知识
2. AI 自动生成标题、摘要、标签、分类、重要程度
3. 用户可以确认入库
4. 知识库列表可以查看、搜索、筛选
5. 用户可以基于知识库提问
6. 回答必须带引用来源

## 项目执行规范

本项目是 AI 知识库问答系统。Codex 执行任务时必须遵守以下规则。

### 基本原则

1. 不要重构，除非用户明确要求。
2. 不要修改与本次任务无关的页面。
3. 不要误改侧边栏、顶部搜索框、底部输入框、用户问题气泡、历史问答区域。
4. 不要删除已有功能。
5. 不要创建未被使用的新组件。
6. 修改前先定位真实生效的组件。
7. 修改后必须检查 git diff。
8. 不要自动 git commit。
9. 不要自动 git push。
10. 完成后只告诉用户是否可以提交。

### 成功功能冻结 / 只改需求范围

1. 已经成功跑通、验收通过或线上可用的功能默认冻结，绝对不要乱动、重构、回滚或重复开发。
2. 每个 Codex 线程只能修改自己负责的明确需求模块，不允许跨模块修改。
3. 只允许修改用户本次明确提出的需求位置，不允许改动无关文件、无关页面、无关 API 或无关配置。
4. 修改前必须先检查真实生效文件、调用链和当前 git 状态，再做最小化修改。
5. 严格保护阿里云后端部署、登录、卡密、管理员端、数据库/Prisma、APK/EXE 登录、文字同步、user-test 强制更新、user-sync-test 临时同步测试通道、图片/文件/拍照附件同步、已成功 UI/上传/历史功能。
6. 严格保证 Tailwind/CSS 样式正常加载，不得破坏 `globals.css` 和 `app/layout.tsx` 的全局样式导入。
7. 如果需求可能影响冻结功能，必须先说明风险并等待用户明确确认后再修改。

### 主控线程与 Worktree 线程边界

1. 主控线程默认只负责检查 `git status`、查看 diff、合并分支、运行 verifiers、部署阿里云、发布 GitHub Release，不直接开发新功能。
2. 任何开发、修复、UI 调整、API 调整都必须交给对应 Worktree 线程执行；主控线程只做审核、合并和发布。
3. 每个 Worktree 线程只能修改自己负责的模块，不允许跨模块顺手修、顺手重构或顺手清理。
4. 合并顺序固定为：后端 API / 统计 -> 管理员投喂版 -> 超级管理员后台 -> 用户端 UI -> APK / EXE / 更新发布 -> 测试修复。
5. 合并前必须检查 diff 范围、lint/typecheck/build、Prisma 相关状态和冻结功能风险。
6. 不允许 `git add .`；只能显式 add 本次需求相关文件。
7. 不允许把 APK、EXE、ZIP、build、release、windows-runtime、node_modules、`.next` 等构建产物提交到 Git。

### 冻结保护清单

除非用户明确点名要求修改，否则以下能力视为冻结成功功能：

1. 阿里云后端部署结构、Nginx/PM2/API_BASE_URL 线上访问链路。
2. 登录、注册、卡密激活、鉴权、用户权限判断。
3. 数据库 schema、Prisma schema、migration、seed、生产数据。
4. 管理员端、超级管理员端、ingest、chat、upload 管理逻辑。
5. APK / EXE 登录、同账号文字同步、历史记录同步。
6. 图片 / 文件 / 拍照附件上传、展示、云端 URL 同步主链路。
7. `user-test` 强制更新通道、Raw manifest、GitHub Release 测试资产。
8. `user-sync-test` 临时同步测试通道及其历史资产。
9. 已验收的用户端 UI、上传等待区、历史抽屉、设置面板、AI Markdown/GPT 风格回复。
10. Tailwind/CSS 全局样式加载，尤其是 `globals.css` 和 `app/layout.tsx` 的导入关系。

### 修改前检查与最小修改

1. 修改前必须先定位真实生效文件、调用链、接口路径、当前分支和当前 dirty 文件。
2. 如果工作区已有无关 dirty 文件，必须忽略并保护，不得格式化、覆盖或回滚用户改动。
3. 修复问题时优先补兼容、兜底和错误提示；不要为了局部问题重写整套流程。
4. 如果发现需求必须触碰冻结清单，先停止并说明原因、风险和建议拆分方案。
5. 完成后必须输出：修改文件、测试命令、验收结果、风险点、是否可以提交。

### Review 与 Verifiers 要求

完成每次修改后，必须进行代码自查和验证。

#### 一、diff 自查

请执行或检查：

1. git status
2. git diff --stat
3. git diff

自查要求：

1. 确认只修改了本次任务相关文件。
2. 确认没有误改无关 UI。
3. 确认没有删除已有功能。
4. 确认没有引入重复组件。
5. 确认没有把测试代码错误带入生产逻辑。
6. 如果发现无关修改，必须还原无关修改。

#### 二、运行 verifiers

请按项目实际情况运行：

1. npm run lint
2. npm run typecheck
3. npm run build

如果某个命令不存在，请说明“不存在，已跳过”，不要因此乱改 package.json。

如果本次涉及测试文件或已有测试命令，请运行相关测试。

#### 三、专项 review

每次修改后请检查：

1. 是否满足本次需求。
2. 是否改动范围过大。
3. 是否破坏现有功能。
4. 是否有 TypeScript 类型风险。
5. 是否有移动端溢出风险。
6. 是否有复制按钮失效风险。
7. 是否有 customerAnswer / finalAnswer 兼容问题。
8. 是否有 debugMarkdown=1 泄露到正式页面的问题。
9. 是否有性能问题。
10. 是否有未处理的异常分支。

#### 四、最终回复格式

完成后请按下面格式回复：

1. 修改文件：
- 文件 A：改了什么
- 文件 B：改了什么

2. Verifiers 结果：
- npm run lint：通过 / 失败 / 不存在
- npm run typecheck：通过 / 失败 / 不存在
- npm run build：通过 / 失败 / 不存在
- 其他测试：通过 / 失败 / 不存在

3. Review 结果：
- 是否发现无关修改：是 / 否
- 是否发现潜在问题：是 / 否
- 如有问题，说明是什么

4. 是否可以提交：
- 可以 / 不建议
- 原因

最终回复还必须明确包含：修改文件、测试命令、验收结果、风险点。
